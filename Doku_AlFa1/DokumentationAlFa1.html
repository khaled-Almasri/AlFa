Übungsblatt 08 – DTO & MapStruct

Ziel

Ziel dieses Übungsblatts war die saubere Trennung zwischen Persistenzschicht (Entity) und REST-API mittels DTOs. Außerdem wurde MapStruct zur automatischen Objektumwandlung eingesetzt.

DTO-Konzept

Für die REST-Schnittstelle werden eigene Data Transfer Objects (DTOs) verwendet, um interne Entity-Strukturen nicht direkt nach außen zu geben.

  • Person – JPA Entity (Datenbank)
  • PersonDTO – Ausgabe über die REST-API
  • PersonCreateDTO – Eingabe beim Erstellen neuer Personen

MapStruct Mapper

Die Umwandlung zwischen Entity und DTO erfolgt mit MapStruct. Der Mapper wird automatisch von Spring verwaltet. 


@Mapper(componentModel = "spring")
public interface PersonMapper {

    PersonDTO toDto(Person person);

    List<PersonDTO> toDtoList(List<Person> persons);
}

Automatisch generierter Code

MapStruct erzeugt zur Compile-Zeit eine Implementierung des Mappers im Verzeichnis target/generated-sources.

Dieser Code wird automatisch generiert und darf nicht manuell verändert werden.

Ergebnis

  • Saubere Trennung von Entity und API-Modell
  • Keine direkte Rückgabe von JPA-Entities
  • Automatische und typsichere Mapping-Logik durch MapStruct

Übungsblatt 09 – Validierung von REST-Requests

Ziel

Ziel dieses Übungsblatts war es, die serverseitige Validierung von REST-Anfragen mit @Valid und Bean Validation umzusetzen. Ungültige Eingaben sollen vom Server automatisch abgelehnt werden.

Umsetzung

Validierung im Controller

Der POST-Endpunkt zur Erstellung einer Person verwendet die Annotation @Valid, um eingehende Requests zu prüfen. 


@PostMapping("/api/persons")
public ResponseEntity<PersonDTO> createPerson(
        @Valid @RequestBody PersonCreateDTO dto) {
    ...
}

Validierungsregeln im DTO

Die Validierungsregeln sind im PersonCreateDTO definiert: 


public record PersonCreateDTO(
    @NotBlank String firstName,
    @NotBlank String lastName,
    @Email @NotNull String email
) {}
  • @NotBlank verhindert leere Namen
  • @Email stellt ein gültiges E-Mail-Format sicher
  • @NotNull verhindert fehlende Werte

Test der Validierung

Zur Überprüfung der Validierung wurde eine ungültige POST-Anfrage an /api/persons gesendet (leere Namen und ungültige E-Mail).

Der Server antwortete korrekt mit HTTP 400 – Bad Request. Damit ist nachgewiesen, dass die Validierung mit @Valid funktioniert.

Zur Überprüfung der REST-Endpunkte wurde zusätzlich eine .http-Datei im Projekt angelegt. Diese ermöglicht es, HTTP-Anfragen (GET, POST) direkt in IntelliJ auszuführen, ohne den Browser oder externe Tools zu verwenden.

Mithilfe der HTTP-Datei konnten sowohl gültige als auch ungültige Requests getestet werden, um sicherzustellen, dass die @Valid-Validierung korrekt funktioniert (z. B. 201 Created bei gültigen Daten und 400 Bad Request bei ungültigen Daten).


Ergebnis

  • Ungültige Eingaben werden mit HTTP 400 abgelehnt
  • Gültige Eingaben führen zu HTTP 201 (Created)
  • Die serverseitige Validierung funktioniert wie erwartet

Fazit

In den Übungsblättern 01–09 wurde ein vollständiges Spring-Boot-Backend mit Docker, Datenbank, REST-API, DTO-Mapping, .HTTP und sauberer Architektur umgesetzt.

Ziel dieses Übungsblatts war die Implementierung eines REST-Endpunkts zum teilweisen Aktualisieren von Personen mittels der HTTP-Methode PATCH. Dabei dürfen ausschließlich Vor- und Nachname geändert werden.

Umsetzung

  • Einführung eines eigenen PersonUpdateDTO für Update-Requests
  • Validierung der Eingaben mit @Valid und Bean Validation
  • Umsetzung der Update-Logik in einem separaten PersonUpdateService
  • Verwendung von MapStruct mit @MappingTarget für partielle Updates
  • Schutz unveränderlicher Felder (id, uuid, email)

REST-Endpunkt


PATCH /api/persons/{uuid}

Die UUID der zu ändernden Person wird über die URL übergeben, die neuen Werte über den Request-Body.

Fehlerbehandlung

Zur sauberen Behandlung von Fehlerfällen wurde ein globaler Exception-Handler (@RestControllerAdvice) implementiert. Dadurch werden interne Fehler vermieden und konsistente HTTP-Statuscodes zurückgegeben.

  • Ungültige Eingaben → HTTP 400 Bad Request
  • Nicht existierende UUID → HTTP 400 Bad Request
  • Erfolgreiches Update → HTTP 200 OK

Tests

Die Funktionalität wurde über eine .http-Datei mit dem IntelliJ HTTP Client getestet. Dabei wurden sowohl erfolgreiche als auch fehlerhafte PATCH-Anfragen überprüft.


Übungsblatt 11

Ziel

In Übungsblatt 11 wurde das Löschen von Personen über eine REST-API mittels der HTTP-Methode DELETE umgesetzt.

Umsetzung

In der Repository-Schicht wurden Methoden zum Finden und Löschen von Personen anhand ihrer UUID definiert. Das Löschen erfolgt über einen eigenen Service, der mit @Transactional annotiert ist und vorab prüft, ob die Person existiert.

Der DELETE-Endpunkt /api/persons/{uuid} liefert bei erfolgreichem Löschen den Status 204 (NO CONTENT). Existiert die Person nicht, wird über den globalen Exception-Handler der Status 404 (NOT FOUND) zurückgegeben.

Tests

Die Funktionalität wurde mit HTTP-Tests überprüft: nicht existierende Person (404), erfolgreiches Löschen (204) sowie erneutes Löschen derselben Person (404).

Ziel

Ergebnis

Beim Start der Anwendung werden Beispielpersonen und ein Raum automatisch angelegt. Die Datenbankstruktur bildet Personen, Räume und Reservierungen korrekt ab.

Dokumentation – Übungsblatt 13

Ziel

Ziel von Übungsblatt 13 ist die Implementierung einer REST-Funktionalität zur konkreten Reservierung von Räumen.

Umsetzung

Für Reservierungen wurden eigene DTOs erstellt, die ausschließlich die UUIDs der referenzierten Person und des Raumes enthalten. Dadurch bleiben die DTOs lose gekoppelt und übersichtlich.

Die Umwandlung zwischen Entitäten und DTOs erfolgt über MapStruct-Mapper. Die Zuordnung der referenzierten UUIDs wird explizit konfiguriert.

In der Service-Schicht wird vor dem Speichern einer Reservierung geprüft, ob die angegebene Person und der Raum existieren. Ist dies nicht der Fall, wird eine entsprechende Exception ausgelöst.

Über einen REST-Controller können Reservierungen per POST /api/reservations angelegt werden. Bei erfolgreicher Erstellung wird der HTTP-Status 201 (Created) zurückgegeben.

Tests

Die Funktionalität wurde mit HTTP-Tests überprüft:

  • Erfolgreiche Reservierung mit existierender Person und Raum → 201
  • Reservierung mit unbekannter Person oder unbekanntem Raum → 404

Ergebnis

Die Reservierungsfunktion arbeitet korrekt, prüft die Existenz der referenzierten Entitäten und liefert passende HTTP-Statuscodes.
